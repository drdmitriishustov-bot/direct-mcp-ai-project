# Article Writer — написание и публикация статей

Полный цикл: тема — семантика — текст — картинка — анти-AI — публикация на Lidfly.

**MCP-серверы:**
- `yandex-direct` — Wordstat (`wordstat_top_requests`, `wordstat_dynamics`)
- `lidfly` — публикация статьи + генерация картинок (`generate_ad_image`, Gemini Flash / nano banana 2)

**Юридическая проверка:** перед финальной публикацией сверься с `LEGAL.md`.

---

## Workflow

Строгая последовательность шагов. На каждом шаге 2-5 — **останавливайся и спрашивай пользователя**, не иди дальше без подтверждения.

### Шаг 1. Тема статьи

Если тема не передана в промпте — спросить у пользователя:

> О чём статья? Сформулируй тему и, если есть, аудиторию/угол подачи.

После получения темы — сразу переходи к шагу 2.

### Шаг 2. Семантическое ядро (Wordstat)

1. **Собрать базовые фразы** по теме (3-5 вариантов) — синонимы, близкие запросы, коммерческие и информационные варианты.

2. **Частотность через MCP:**
   ```
   wordstat_top_requests(phrase: "<базовая фраза>")
   ```
   Для каждой базовой фразы — получить до 128 связанных запросов с частотностью.

3. **Отсеять мусор:**
   - Частотность < 30 — выкинуть
   - Явные чужие бренды, географические лишние, нерелевантные — выкинуть
   - Информационные («что такое», «отзывы», «как», «почему») — оставить, они как раз для статьи

4. **Сгруппировать по смыслу** и вывести пользователю таблицу (топ 20-30 самых релевантных):

   | № | Фраза | Частотность | Интент |
   |---|-------|-------------|--------|
   | 1 | как выбрать X | 5 400 | информационный |
   | 2 | X для начинающих | 2 100 | информационный |
   | ... | ... | ... | ... |

5. **Спросить пользователя:**

   > Вот семантика по теме. Какие фразы использовать в статье? (номера через запятую, «все», или укажи свои)
   >
   > Минимальный объём статьи — 8000 символов. Оставить 8000 или другой?

6. **Дождаться ответа.** Зафиксировать выбранные фразы и минимальный объём.

### Шаг 3. Написание статьи

После подтверждения ключевых и объёма:

1. **Структура статьи** (обязательно):
   - **H1** — заголовок с главной ключевой фразой
   - **Лид** (1-2 абзаца, 150-250 символов) — суть статьи, ключевик в первых 100 словах
   - **4-8 блоков H2** — раскрывают тему по подтемам
   - **H3** — внутри H2, где нужна детализация
   - **Итог / выводы** — короткий summary + CTA (если уместно)
   - **FAQ** (опционально, 3-5 вопросов) — для featured snippets

2. **Правила текста:**
   - Объём не менее указанного минимума (по умолчанию 8000 символов **без пробелов**)
   - Ключевые фразы распределены естественно, плотность 1-2%
   - Главная фраза — в H1, первом абзаце, одном из H2, заключении
   - Длина предложений разная — от 5 до 25 слов
   - Списки, таблицы, примеры — где уместны
   - Никаких клише «в современном мире», «важно отметить», «следует подчеркнуть»

3. **Юридическая проверка:** сверить текст с `LEGAL.md` (запрещённые термины, «лучший», «№1», гарантии без оснований). Если есть замечания — исправить.

4. **Сохранить черновик** в `data/articles/<slug>-draft.md` (создать папку если нет). Это оригинальная чистая версия с разметкой.

5. **Показать пользователю** краткое превью (заголовок + структура H2) и сказать:

   > Черновик готов: `<путь>`. Считаю символы: XXXX (без пробелов). Переходить к генерации картинки?

6. **Дождаться подтверждения** перед следующим шагом.

### Шаг 4. Промпт для картинки

Генерация — через `generate_ad_image` (Gemini Flash Image Preview / nano banana 2), доступен на эндпоинте `lidfly` (а также `yandex-direct` и `vk-ads` — это shared-инструмент).

**Квота:** 5 бесплатных вызовов/месяц, далее 6 ₽ с баланса **за вызов** (не за файл). Один вызов может отдать до 6 файлов разных пропорций через `crop_to`.

1. **Сгенерировать промпт** на русском (Gemini понимает). Формула: **объект + стиль + фон + освещение + палитра**. Конкретика, а не общие слова.

   Пример структуры:
   ```
   <главный объект с деталями>, <стиль: фотореализм / минимализм / иллюстрация>,
   <фон и окружение>, <освещение>, <цветовая палитра>,
   главный объект по центру композиции
   ```

   **Не проси текст/надписи на картинке** — кириллица рисуется плохо. Заголовок статьи пойдёт через настройки публикации, не на обложку.

2. **Перед генерацией — проверить, нет ли уже свежей:**
   ```
   get_recent_images({ limit: 10 })
   ```
   Read-only, без списаний. Если подходящий public_url уже есть — используй его, не генери заново.

3. **Показать пользователю:**

   > Промпт для обложки:
   > ```
   > <промпт>
   > ```
   >
   > Формат обложки: `lidfly_hero_1920x1080` (16:9, 1920×1080).
   > Генерировать? (да / нет / править промпт / добавить доп. пропорции)

4. **Дождаться ответа.**
   - «нет» — пропустить шаг 5, перейти к шагу 6
   - «править» — переработать промпт и снова спросить
   - «добавить пропорции» — уточнить какие (`vk_square_1080`, `vk_story_1080x1920` и т.п.) и добавить в `crop_to`
   - «да» — шаг 5

### Шаг 5. Генерация картинки (если подтвердили)

1. **Вызвать `generate_ad_image`** на эндпоинте `lidfly`:

   ```
   generate_ad_image({
     prompt: "<согласованный промпт>",
     format: "lidfly_hero_1920x1080",
     crop_to: [<опциональные доп. пропорции, если просил пользователь>]
   })
   ```

   **Правила:**
   - Один вызов = одно списание, независимо от числа файлов в результате
   - Каждый `crop_to` должен помещаться в `format` по обеим осям — если просили story 9:16 + hero 16:9, бери базой `vk_story_1080x1920` и добавь hero в `crop_to`
   - Для `custom` формата — указать `width` и `height` (256–2048)

2. **Таймаут 504 — НЕ ретрай вслепую.** Картинка могла успеть сохраниться, повтор = двойное списание.
   - Сначала `get_recent_images({ limit: 5 })`
   - Если свежая запись с нужным форматом и `batch_id` есть — взять её `public_url`
   - Только если нет — повторить `generate_ad_image`

3. **Забрать `public_url`** из ответа. Для статьи на Lidfly можно:
   - отдать `public_url` напрямую в поле обложки при публикации статьи (шаг 7)
   - либо положить в assets сайта: `lidfly_upload_image(subdomain, image_url: "<public_url>")` — если нужен хостинг в домене сайта

4. **Сохранить локальную копию** (опционально, для архива): скачать по `public_url` в `data/articles/<slug>-cover.<ext>`.

5. **Показать пользователю:**

   > Обложка готова: `<public_url>`
   > (локальная копия: `<slug>-cover.<ext>`)
   >
   > Устраивает или перегенерить с другим промптом?

6. **Дождаться подтверждения** перед публикацией.

### Шаг 6. Удаление AI-маркеров

Применить скилл `ai-markers-remove` к тексту статьи:

- 2-5 орфографических мини-ошибок
- 5-10 пунктуационных ошибок (чаще пропуск запятой перед «что», «который», «если»)
- Убрать клише и «идеальные» AI-переходы
- Разнообразить длину предложений
- Добавить 1-2 разговорных оборота, субъективных оценок
- Сохранить смысл, разметку (H1/H2/H3/списки) и структуру

**Сохранить** переработанную версию в `data/articles/<slug>-final.md`. Оригинальный `-draft.md` не трогать — это бэкап.

**Не показывать пользователю весь текст** (длинно) — только сообщить:
> AI-маркеры убраны, финальный файл: `<путь>`. Публикую на Lidfly.

### Шаг 7. Публикация на Lidfly

1. **Проверить авторизацию** — если MCP `lidfly` требует OAuth, провести пользователя через flow.

2. **Найти инструмент публикации** — в списке инструментов `lidfly` найти создание статьи/поста (ключевые слова: `create`, `post`, `article`, `publish`).

3. **Опубликовать:**
   - `title`: H1 статьи
   - `content`: содержимое `<slug>-final.md` (с Markdown-разметкой, если поддерживается)
   - `cover_image`: `public_url` обложки (если была сгенерирована)
   - `tags` / `keywords`: выбранные ключевые фразы из шага 2
   - `description` / `meta_description`: 150-160 символов из лида статьи

4. **Получить URL** опубликованной статьи из ответа API.

5. **Отдать пользователю:**

   > Статья опубликована: `<URL>`
   > Файл оригинала: `<slug>-draft.md`
   > Файл финала: `<slug>-final.md`
   > Обложка: `<slug>-cover.<ext>` (если генерили)
   >
   > Проверяй и дай знать, нужны ли правки.

## Чеклист перед публикацией

- [ ] Тема подтверждена пользователем
- [ ] Ключевые фразы выбраны (шаг 2.5)
- [ ] Объём не менее минимума (8000 без пробелов по умолчанию)
- [ ] H1 с главным ключевиком, структура H2/H3
- [ ] Ключевик в первом абзаце
- [ ] Юр. проверка по `LEGAL.md` пройдена
- [ ] Черновик сохранён в `data/articles/<slug>-draft.md`
- [ ] Картинка согласована (или пропущена)
- [ ] AI-маркеры убраны (финал в `<slug>-final.md`)
- [ ] Публикация успешна, URL получен

## Файлы и артефакты

На выходе в `data/articles/` по каждой статье:

| Файл | Содержимое |
|------|------------|
| `<slug>-draft.md` | Оригинальный чистый черновик с разметкой |
| `<slug>-final.md` | Версия после удаления AI-маркеров (ушла на публикацию) |
| `<slug>-cover.<ext>` | Сгенерированная обложка (опционально) |

`<slug>` — короткая транслитерация темы, kebab-case (например, `kak-vybrat-crm-2026`).

---

## Когда НЕ использовать скилл

- Короткие тексты (< 2000 символов) — объявления, посты соцсетей, этот workflow избыточен
- Технические статьи в репу (README, docs) — не нужна публикация на Lidfly
- Рекламные тексты для Директа — используй скилл `yandex-direct-campaign-builder`
