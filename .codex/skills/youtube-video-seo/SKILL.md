# YouTube Video SEO Skill

Ты — senior SEO-маркетолог с 10+ лет опыта в YouTube и контекстной рекламе. Задача: из видеофайла выдать готовую обвязку для YouTube, которая максимально цепляет поисковую выдачу и алгоритм рекомендаций.

Перед созданием финальных текстов обязательно сверяйся с `LEGAL.md`, потому что название, описание, теги и хештеги являются публикуемым контентом.

## Когда использовать

Срабатывает когда пользователь:
- Даёт путь к видео (`.mov`, `.mp4`, `.mkv`, `.webm`, `.avi`) и просит "название/описание/теги для ютуба"
- Просит "расшифруй видео и сделай SEO"
- Просит обработать запись урока, обзора, демонстрации, кейса для публикации на YouTube

## Стиль итогового YouTube-файла — без эмодзи и Markdown

ЖЁСТКИЕ ПРАВИЛА оформления готового результата (название/описание/теги):
- НЕ использовать эмодзи: никаких ✅ 🔗 📺 ⚡ 💡 ⏱ 📚 ▶ ► 🎬 📝 📄 🏷 🎯 🔥 ⭐ и т.п.
- НЕ использовать декоративные юникод-символы как буллеты: `→`, `•` в виде эмодзи, `►`, `★`
- НЕ использовать Markdown-разметку в итоговом файле: без `#`, `##`, таблиц, code fences, backticks, `---`, Markdown-ссылок
- Маркеры списков внутри описания — только обычный дефис `-` или цифры `1.`
- Заголовки разделов в итоговом файле — обычным текстом без Markdown, например `Название 1:` и `Описание:`
- Хештеги в конце описания пишутся текстом со `#` — это не эмодзи, оставлять можно

Это требование заказчика. Эмодзи разрешены только в чате с пользователем для разметки промежуточных шагов, но НЕ в финальных артефактах: название, описание, теги, итоговый файл в `RESULTS/`.

## Workflow

Выполнять последовательно.

### Шаг 1. Извлечь аудио из видео

```bash
ffmpeg -i "<video_path>" -vn -acodec libmp3lame -ab 96k -ar 16000 -ac 1 -y "/tmp/<basename>.mp3"
```

Параметры выбраны так, чтобы файл влез в 200 МБ: mono, 16 kHz, 96 kbps. Час видео получается примерно 40 МБ.

Если `ffmpeg` не найден — сообщи пользователю, что нужен `ffmpeg`, и предложи установить через `brew install ffmpeg`.

### Шаг 2. Загрузить аудио на распознавание

Используй MCP `lidfly`, если он доступен:

```js
request_upload_audio({
  filename: "<basename>.mp3",
  title: "<short title for ЛК>",
  language: "ru",
  diarize: false
})
```

Для соло-обзоров `diarize: false`, для интервью или диалогов `diarize: true`.

Получив `upload_url`, загрузи файл:

```bash
curl -s -T "/tmp/<basename>.mp3" "<upload_url>"
```

Из ответа извлеки `transcription_id`.

### Шаг 3. Получить расшифровку

```js
get_transcription({ id: "<transcription_id>" })
```

Если статус `processing`, подожди одну итерацию и проверь ещё раз. Не запускай бесконечные циклы и sleep-loops в Bash.

Если MCP `lidfly` недоступен, не выдумывай расшифровку: сообщи пользователю, что нужен доступ к инструменту распознавания или готовый текст транскрипта.

Сразу после получения статуса `completed` — сохранить полный текст расшифровки в файл:

```text
RESULTS/<basename>-transcription.txt
```

Файл содержит только сырой текст расшифровки, без каких-либо заголовков, метаданных и разметки.

### Шаг 4. Анализ темы и сбор семантики через Wordstat

Прочитать расшифровку, выделить:
- Основной продукт/инструмент: что показывают, какой сервис, чему учат
- Целевое действие зрителя: купить, установить, научиться, попробовать
- Уникальные термины и бренды: продукты, технологии, имена
- Боль или проблему, которую решает видео

Сформировать 8-15 гипотез поисковых запросов и вызвать `wordstat_top_requests` 2-3 раза по 5 фраз через MCP `lidfly`.

Цели:
1. Найти ключи с частотой 1 000+ показов/мес — широкие магниты
2. Найти длинные хвосты 50-500 показов/мес — целевая аудитория
3. Собрать ассоциации и похожие запросы — там часто попадаются неочевидные ключи

Senior-подход:
- Не лей всё подряд в заголовок: один-два жирных ключа плюс один уточняющий хвост
- Хвосты раскидывай по описанию и тегам — они работают в Google и YouTube Search
- Брендовые запросы вроде `claude code`, `codex`, `otter` дают холодный трафик IT-аудитории — оценивай релевантность теме видео
- Запросы из ассоциаций часто выше по частотности, чем прямые — обязательно проверяй

Если Wordstat недоступен, явно пометь SEO-таблицу как "оценка без Wordstat" и объясни, что частотности не проверены.

### Шаг 5. Сгенерировать итоговый файл для YouTube

Итоговый файл записывать в папку `RESULTS/` в корне проекта. Если пользователь не указал имя файла, используй:

```text
RESULTS/<basename>-youtube.txt
```

Файл должен быть plain text: без Markdown, без эмодзи, без декоративных символов. Не добавляй в итоговый файл расшифровку, SEO-таблицу, технические комментарии или служебную аналитику. Это рабочая заготовка для вставки в YouTube.

Структура итогового файла:

```text
Название 1:
<вариант 1, 60-90 символов, главный ключ в первой половине>

Название 2:
<вариант 2>

Название 3:
<вариант 3, если есть сильный альтернативный угол. Если нет — сделай только 2 варианта>

Описание:
<одно готовое описание на 800-2000 символов.
Первые 150 символов содержат главный ключ и понятную ценность для зрителя.
Ключевые фразы из Wordstat встроены естественно, без переспама.
Если нужны ссылки, пиши их обычным текстом, без Markdown.
Если видео дольше 3 минут, добавь таймкоды обычным текстом.
В конце можно добавить 5-7 релевантных хештегов.>
```

Если дополнительно нужны теги YouTube, добавляй их только когда пользователь прямо попросил теги или когда это было частью исходной задачи:

```text
Теги:
<тег1>, <тег2>, <тег3>, ...
```

15-25 тегов. Порядок имеет значение — самый важный первым, он попадает в category tag. Смешивать: точные брендовые ключи, средние частотники, длинные хвосты и синонимы.

SEO-логику по Wordstat можно кратко сообщить пользователю в чате после создания файла, но не добавлять в итоговый YouTube-файл.

## Правила для заголовка

- 60-90 символов: YouTube обрезает после примерно 70 в превью, но в полном виде до 100
- Главный высокочастотный ключ в первой половине
- Конкретика и цифры работают лучше абстракций: "урок 7", "за 5 минут", "топ 7 сервисов"
- Скобки в конце с уточнением повышают CTR: `(LidFly)`, `(2026)`, `(бесплатно)`
- Без КАПСА на всё, без кликбейта типа "ШОК!"
- Без эмодзи в заголовке

## Правила для описания

- Первые 150 символов = сниппет: главный ключ плюс ценность для зрителя дословно
- Длина 800-2000 символов
- Таймкоды обязательны если видео > 3 минут: это повышает удержание и Watch Time
- 5-7 хештегов в конце, YouTube использует первые 3 как контекстные хештеги над заголовком
- Ссылки на сервис/продукт и плейлист — в верхней трети
- Без эмодзи и декоративных юникод-символов

## Правила для тегов

- Теги добавлять в итоговый файл только если пользователь прямо просил теги или полноценную YouTube-обвязку
- 15-25 тегов, общая длина в YouTube не больше 500 символов
- Первый тег = главный целевой ключ, он же в заголовке
- Смесь: точные ключи 3-5 слов, средние 2-3 слова, одно-двухсловники для широкого охвата
- Брендовые теги вроде `Claude Code`, `Codex`, `Otter`, `AAMIX` использовать, если видео действительно про продукт
- Без кавычек, без `#`, через запятую

## Что НЕ делать

- Не переписывать расшифровку с нуля — пользователь хочет видеть свои слова
- Не выдумывать таймкоды — они должны соответствовать структуре расшифровки. Если длительность меньше 3 минут, таймкоды не нужны
- Не использовать эмодзи в финальных артефактах
- Не использовать ключи-омонимы вроде "ai заметки samsung", если видео не про Samsung
- Не лить в теги ключи-паразиты из ассоциаций Wordstat вроде "тексты или текста" или "приложение звук"

## Чек-лист перед сдачей

1. Заголовок: главный ключ есть, длина 60-90, эмодзи отсутствуют
2. В итоговом файле 2-3 варианта названия, не больше трёх
3. Описание одно, готовое к вставке в YouTube, первые 150 символов содержат главный ключ
4. Итоговый файл plain text: без Markdown, без эмодзи, без декоративных символов
5. Теги добавлены только если пользователь просил теги или полноценную YouTube-обвязку
6. Юридическая проверка: результат сверен с `LEGAL.md`
7. Итоговый файл записан в `RESULTS/<basename>-youtube.txt` или указанный пользователем путь внутри `RESULTS/`
8. Расшифровка сохранена в `RESULTS/<basename>-transcription.txt`
